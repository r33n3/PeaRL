# ADR-001 — Externalize Trust Adjudication to MASS 2.0; Keep PeaRL Non-Agentic

**Status:** Accepted
**Date:** 2026-04-05
**Authors:** PeaRL maintainers

---

## Context

PeaRL currently performs three kinds of work:

1. Governance and approval orchestration
2. Evidence and finding ingestion
3. Built-in static/compliance scanning

PeaRL also integrates with MASS 2.0 as an external scan source. MASS findings are ingested as ordinary findings, and MASS scan completion is represented by marker findings. Promotion decisions depend on gate results, approval mode, and trust accumulation.

This creates an architectural risk:

- Development agents can optimize toward gate passing
- PeaRL may drift into validator behavior if model calls are added to workers
- MASS trust judgments could be blurred into generic findings, losing their semantic distinction
- Auto-pass and auto-approval paths can bypass future trust-review requirements

The intended architecture separates concerns cleanly:

- **PeaRL** — governance and evidence control plane (deterministic, model-free, auditable)
- **MASS 2.0** — validator and trust-adjudication plane (model-based, multi-judge, open model choice)
- **Human reviewers** — final authority for consequential elevation

---

## Decision

**PeaRL will remain the non-agentic governance plane and will not host model-based trust judges, provider orchestration, or model selection logic for promotion trust decisions.**

### MASS 2.0 owns:

- Validator-agent orchestration
- Model and provider selection for trust evaluation
- Multi-judge trust evaluation
- Promotion trust verdict generation
- Confidence scoring and evidence gap analysis

### PeaRL owns:

- Promotion candidate packaging (assembling evidence for MASS review)
- Outbound trust review requests to MASS
- Ingestion and storage of MASS trust verdict artifacts as a distinct data type
- Gate policy requiring MASS review for specified transitions
- Reviewer presentation of trust verdicts alongside other evidence
- Final approval routing and auditability

### Architectural hard constraint:

**PeaRL workers are model-free.** Workers perform deterministic computation only — data transformation, scoring, querying, routing. No LLM calls, no model API calls, no embeddings. This constraint is non-negotiable and applies to all current and future workers.

These schemas define the integration boundary. Both sides must implement against the same schema version.

- `compiled_context_ref` is a pointer to the full compiled agent context package (compiled by the `compile_context` worker). MASS may fetch this for deeper analysis.

### MassTrustVerdictArtifact

Returned by MASS to PeaRL on verdict completion.

```json
{
  "schema_version": "1.0",
  "verdict_id": "vrd_<nanoid>",
  "request_id": "treq_<nanoid>",
  "project_id": "proj_abc123",
  "evaluation_id": "eval_<nanoid>",
  "source_environment": "preprod",
  "target_environment": "prod",
  "recommendation": "approve | conditional | reject",
  "confidence": "high | medium | low",
  "acceptable": true,
  "blocking_concerns": [
    {
      "concern_id": "c1",
      "domain": "excessive_agency | data_handling | transparency | security | fairness | compliance",
      "severity": "blocking | advisory",
      "description": "...",
      "evidence_ref": "<finding_id or report_id if applicable>"
    }
  ],
  "evidence_gaps": [
    {
      "gap": "No recent adversarial test results",
      "impact": "Cannot assess robustness under attack"
    }
  ],
  "constraints": [
    {
      "constraint": "Rate limit agent to 100 requests/min in production",
      "owner": "operator",
      "expires_at": "<iso8601 or null>"
    }
  ],
  "judge_summaries": [
    {
      "judge_id": "judge_1",
      "domain": "security",
      "summary": "...",
      "verdict": "approve | conditional | reject",
      "provider_model_trace": {
        "provider": "anthropic",
        "model": "claude-sonnet-4-6",
        "invocation_id": "<trace id>"
      }
    }
  ],
  "overall_provider_model_trace": {
    "orchestrator_provider": "anthropic",
    "orchestrator_model": "claude-sonnet-4-6",
    "judge_count": 3,
    "total_invocations": 7
  },
  "generated_at": "<iso8601>",
  "expires_at": "<iso8601>",
  "mass_version": "2.0.4"
}
```

**Notes:**
- `recommendation` is MASS's judgment: `approve` (proceed), `conditional` (proceed with constraints), `reject` (do not promote).
- `confidence` is categorical (`high / medium / low`). PeaRL gate rules can require minimum confidence.
- `acceptable` is a boolean convenience field derived from `recommendation != reject`. PeaRL gate rule `MASS_VERDICT_ACCEPTABLE` checks this field.
- `blocking_concerns` with `severity: blocking` are the primary signal for `MASS_BLOCKING_CONCERNS_ZERO` gate rule.
- `judge_summaries[].provider_model_trace` is required — auditability of which model generated each judgment is non-negotiable.
- `expires_at` defines freshness. PeaRL gate rule `MASS_VERDICT_FRESH` checks `generated_at` against `expires_at` and a configurable max-age policy.
- `constraints` are post-approval conditions MASS recommends. PeaRL surfaces them to the reviewer; enforcement is the operator's responsibility.

## Consequences

### Positive

- Clean separation between evidence/control plane and model-based adjudication
- PeaRL remains fully deterministic and auditable — every governance decision traceable without model opacity
- MASS can support open model choice (Anthropic, local Nemotron, other providers) without contaminating PeaRL's governance logic
- Reviewers can distinguish raw evidence (findings, reports) from interpreted trust assessments (MASS verdict)
- Future promotion policy can require independent trust review at any transition
- PeaRL can be audited and certified independently of MASS

### Negative

- Additional integration contract to define and maintain (mitigated by this ADR)
- New data model required in PeaRL for trust verdict artifacts
- Current auto-pass and auto-approval logic must be tightened before trust-review gates are live
- PeaRL built-in static scanning may require boundary documentation as the integration matures

### Risk: MASS unavailability

If MASS is unreachable and a promotion requires a fresh trust verdict, PeaRL **fails closed** — promotion is blocked with state `trust_review_unavailable`. A reviewer can override with explicit justification, which is recorded in the audit trail. This is not a failure mode; it is the correct governance behavior.

---

## Rejected Alternatives

| Alternative | Reason rejected |
|---|---|
| Put trust judges directly in PeaRL | Makes PeaRL agentic and mixes governance with model behavior. Breaks determinism and auditability. |
| Store MASS trust verdicts as findings | Trust adjudication is not the same object type as a scanner finding. Collapsing them corrupts both data types and their query semantics. |
| Let MASS verdict directly approve promotion | Governance authority must remain in PeaRL / human review. MASS provides expert evidence; it does not decide. |
| Keep only compliance gates, no trust layer | Passing compliance controls is not the same as being trustworthy for higher autonomy or environment elevation. These are separate questions. |

## Implementation Backlog

### Epic 1 — Trust Review Contract (prerequisite, no code until schemas agreed)

- [ ] Finalize `PromotionCandidateReviewRequest` schema (both sides agree)
- [ ] Finalize `MassTrustVerdictArtifact` schema (both sides agree)
- [ ] Define freshness TTL policy: default `expires_at` = 7 days, configurable per gate
- [ ] Define `acceptable` threshold: `recommendation != reject` (not confidence-based)
- [ ] Define fail-closed behavior: `trust_review_unavailable` state, reviewer override path

### Epic 2 — PeaRL Data Model

- [ ] `TrustVerdictRow` table with `vrd_` ID prefix
- [ ] `TrustVerdictRepository`: get by id, latest by project/environment, latest acceptable, freshness check
- [ ] Alembic migration `006_add_trust_verdict_table.py`
- [ ] No findings table involvement — trust verdicts are a distinct data type

### Epic 3 — Promotion Integration

- [ ] Promotion candidate packaging service: derive candidate from PeaRL evidence
- [ ] `POST /projects/{id}/trust-review/request` — packages candidate, sends to MASS, stores request record
- [ ] `POST /projects/{id}/trust-review/ingest` — receives MASS verdict, validates schema, stores artifact
- [ ] `GET /projects/{id}/trust-review/latest` — latest verdict for project/environment/freshness

### Epic 4 — Gate Policy

- [ ] `MASS_VERDICT_PRESENT` — a verdict exists for this project/environment
- [ ] `MASS_VERDICT_FRESH` — verdict `generated_at` within freshness TTL
- [ ] `MASS_VERDICT_ACCEPTABLE` — `acceptable == true`
- [ ] `MASS_BLOCKING_CONCERNS_ZERO` — no blocking-severity concerns
- [ ] Policy mapping: require MASS review for `preprod → prod` by default
- [ ] Auto-pass blocked when required verdict is absent, stale, or unacceptable

### Epic 5 — Reviewer UX

- [ ] Approval thread: MASS verdict section separate from findings section
- [ ] Show: `recommendation`, `confidence`, `blocking_concerns`, `constraints`, freshness
- [ ] Elevation/release reports include MASS trust review status as first-class section
- [ ] Do not bury verdict in generic risk summaries

### Epic 6 — Approval Safety

- [ ] `auto_pass` blocked when trust review required and verdict absent/stale/unacceptable
- [ ] `approval_mode == auto` cannot bypass required MASS trust review
- [ ] Gate evaluator priority order: behavioral drift → MASS verdict check → auto-pass evaluation

### Epic 7 — Boundary Clarification

- [ ] Review and classify PeaRL built-in scanning: document as static/compliance analyzer, not trust tribunal
- [ ] Add explicit comment block to each worker: "This worker is model-free — no LLM calls"

## Transition Scope

**Transitions requiring MASS trust review (initial policy):**

| Transition | MASS Required | Notes |
|---|---|---|
| dev → pilot | No | Routine iteration |
| pilot → preprod | No (recommended) | Gate rules can be added per project |
| preprod → prod | **Yes** | Fresh verdict required, blocking concerns zero |
| Any → prod with active exceptions | **Yes** | Exceptions do not substitute for trust review |
| Autonomy expansion (allowance profile relaxation) | **Yes** | Elevation of agent scope requires trust review |

---

## What to Watch

- Do not store MASS verdicts as findings
- Do not let PeaRL grow model or provider logic
- Do not let auto-pass bypass trust review when required
- Do not collapse compliance and trust into one status field
- `provider_model_trace` is required in every judge summary — never optional
